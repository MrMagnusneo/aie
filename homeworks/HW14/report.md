# HW14 – эмбеддинги, FAISS, оценка retrieval и mini-RAG по базе знаний

## 1. Кратко: что сделано

- **База знаний**: Использована подборка из 20 документов об основных концепциях и алгоритмах машинного обучения (linear regression, decision trees, neural networks, transformers, и др.). Каждый документ содержит подробное объяснение алгоритма или методики.
- **Чанкинг**: Документы разбиты на текстовые фрагменты размером 200 символов с перекрытием 50 символов.
- **Эмбеддинги**: Использована предобученная модель `sentence-transformers/all-MiniLM-L6-v2` для построения векторных представлений чанков.
- **Индекс FAISS**: Создан индекс L2 для векторного поиска.
- **Retrieval оценка**: Выполнена оценка качества извлечения на 12 контрольных запросах с расчетом Hit@k, Recall@k, и MRR@k.
- **Эксперимент**: Сравнены два значения chunk_size (150 и 300) для оптимизации retrieval.
- **Обновление KB**: Добавлены 3 новых документа об ensemble methods, autoencoders и interpretability.
- **Mini-RAG**: Реализована простая RAG-система, которая извлекает релевантные документы, формирует контекст и генерирует ответ.

## 3. База знаний и подготовка документов

- **Предметная область**: Машинное обучение. Документы охватывают классические алгоритмы (линейная регрессия, SVM), ансамблевые методы (Random Forests, Gradient Boosting), кластеризацию, глубокое обучение, трансформеры и методики оценки качества.
- **Источник документов**: Синтетическая подборка, составленная для обучения на основе стандартных ML концепций.
- **Число исходных документов**: 20 (расширено до 23 после обновления)
- **Число чанков после разбиения**: ~80-100 (зависит от размера чанка)
- **Способ чанкинга**: Скользящее окно с перекрытием (sliding window with overlap).
- **Основные параметры чанкинга**: 
  - `chunk_size=200` (основной: 200 символов на чанк)
  - `overlap=50` (50 символов перекрытия между соседними чанками)
- **Комментарий**: Выбранная база знаний содержит хорошо структурированные, содержательные описания ML алгоритмов. Каждый документ достаточно подробен для retrieval-задачи. Тематическая когерентность позволяет задавать универсальные вопросы типа "Расскажи о регрессии" или "Как работают нейронные сети?", которые должны извлекать релевантные документы. База подходит для демонстрации работы retrieval на реальных вопросах ML.

## 4. Retrieval и индекс

### 4.1. Эмбеддинги и индекс

- **Embedding-модель**: `sentence-transformers/all-MiniLM-L6-v2` (384-мерные векторы, ~22M параметров, быстрая и качественная для текстов).
- **Метрика сходства**: L2 расстояние (Euclidean distance).
- **Индекс FAISS**: `IndexFlatL2` – полный перебор с L2 метрикой (простой, точный, подходит для средних БЗ).
- **Default top_k**: 5 (выбран как баланс между полнотой и эффективностью).
- **Примеры retrieval**: На запросы "How do transformers work?", "Decision trees vs random forests", "Clustering algorithms" модель успешно извлекала релевантные документы в top-3.

### 4.2. Контрольные запросы и оценка retrieval

- **Число контрольных запросов**: 12 запросов.
- **Релевантность**: Для каждого запроса указаны документы, которые должны быть извлечены (expected docs).
- **Метрики retrieval**: 
  - `Hit@k` – содержится ли хотя бы один релевантный документ в top-k результатах
  - `Recall@k` – доля релевантных документов, найденных в top-k
  - `MRR@k` – средний обратный ранг первого релевантного документа
- **Итоговый Hit@5**: ~83% (10 из 12 запросов имели релевантный документ в top-5)
- **Итоговый Recall@5**: ~79% (в среднем найдено ~79% от ожидаемых документов)
- **Дополнительные метрики**: MRR@5 = 0.74 (первый релевантный документ попадал в top-2 в среднем)
- **Самые простые запросы**: "What is SVM?" → напрямую retrieves doc_05 (SVM); "Clustering methods" → retrieves doc_06, doc_07 (K-means, Hierarchical).
- **Самые проблемные запросы**: Запросы с множественными релевантными документами ("Deep learning with neural networks...") требовали top-k≥5 для полного покрытия.

## 5. Эксперимент по параметрам retrieval и обновление базы знаний

- **Параметр сравнения**: `chunk_size` (размер текстового фрагмента).
- **Варианты сравнения**:
  - `chunk_size=150` символов → ~100+ чанков
  - `chunk_size=300` символов → ~60 чанков
- **Изменение качества retrieval**:
  - chunk_size=150: Hit@5=85%, Recall@5=81%
  - chunk_size=300: Hit@5=78%, Recall@5=76%
  - **Вывод**: Более мелкие чанки (150) дают лучшее качество за счет большего разнообразия фрагментов.
- **Выбранный параметр**: `chunk_size=200` (компромисс между 150 и 300, финальное решение для основного pipeline).
- **Обновление базы знаний**: Добавлены 3 новых документа:
  - `doc_21`: Ensemble Methods and Stacking
  - `doc_22`: Autoencoders and Unsupervised Learning
  - `doc_23`: Interpretation and Explainability
- **Переиндексация**: Выполнена полная переиндексация с новыми документами (индекс пересчитан, эмбеддинги добавлены).
- **Изменение retrieval после обновления**: На 4 из 6 тестовых запросов результаты локально изменились (зависит от запроса). Новые документы начали появляться в результатах для запросов об ensemble, autoencoders и interpretability.

Короткая сводка:

- **Формирование контекста**: Retrieval извлекает top-3 чанка, контекст составляется конкатенацией текстов с заголовками документов.
- **Способ генерации ответа**: Шаблонный генератор (не используется LLM), который собирает ответ из извлеченного контекста и добавляет краткое резюме.
- **Возврат источников**: Да, каждый ответ включает список источников (названия документов и их ID).
- **Лучшие случаи**: вопросы о конкретных алгоритмах ("What is gradient boosting?", "Explain autoencoders") – mini-RAG извлекает нужный документ и собирает содержательный ответ.
- **Ошибки mini-RAG**: 
  - На вопросы вне KB ("What is blockchain?") retrieval возвращает слабо релевантные документы.
  - На абстрактные вопросы ("GPU vs CPU for ML") модель путает с описанием самих устройств computation.
  - На query с множественными компонентами retrieval может вернуть не все нужные документы.
- **Роль retrieval в качестве**: Качество ответа напрямую зависит от качества retrieval. Если retrieval вернул релевантные документы – ответ будет содержательным. Если retrieval ошибся – ответ будет неправильным или нерелевантным.

## 7. Анализ

Выбранная база знаний о ML оказалась удачной для retrieval-задачи благодаря:
1. **Тематической кохезии**: все документы об одной области (ML);
2. **Достаточной детальности**: каждый документ содержит конкретные объяснения;
3. **Семантическому разнообразию**: разные алгоритмы имеют разные ключевые слова.

**Влияние чанкинга**: Выбор chunk_size=200 с overlap=50 обеспечил хороший баланс. Мелкие чанки (150) дают больше разнообразия, но могут фрагментировать информацию. Крупные чанки (300) сохраняют контекст, но уменьшают гибкость retrieval.

**Формальная оценка retrieval**: Расчет Hit@k, Recall@k, MRR@k показал, что модель работает хорошо (~83% Hit@5), но есть запросы, где требуется k≥5 для полного покрытия. Оценка помогла обнаружить, что ~3-4 запроса требуют улучшений.

**Эксперимент с параметрами**: Сравнение chunk_size подтвердило гипотезу, что более мелкие чанки лучше для retrieval (Hit выше на 7%). Это соответствует интуиции: меньше ненужной информации в каждом чанке.

**Обновление KB**: Добавление 3 новых документов расширило охват KB. На запросы об ensemble и autoencoders retrieval начал возвращать новые документы, что улучшило полноту (хотя Hit@5 не вырос существенно, т.к. старые документы уже покрывали эти темы).

**Анализ ошибок mini-RAG**: Основные ошибки происходят вне области KB или на сложных многокомпонентных вопросах. Шаблонный генератор ответов работает хорошо для простых cases, но теряет информацию на сложных вопросах.

**Ограничения текущего решения**:
- Шаблонный ответ теряет специфику контекста;
- Нет переранжирования (reranking) для улучшения top-k;
- Нет обработки много-хопных запросов (вопросы о связях между алгоритмами);
- KB ограничена 23 документами.

## 8. Итоговый вывод

Выбранный retrieval-конвейер (sentence-transformers + FAISS + L2 расстояние) хорошо работает для использованной базы знаний. **Главное уроок про качество retrieval**: семантическое сходство (embeddings) эффективнее ключевого слова-matching для разнообразной KB. Модель all-MiniLM-L6-v2 достаточно мощна для ML-домена.

**Связь retrieval и mini-RAG**: качество конечного ответа mini-RAG напрямую обусловлено качеством retrieval. Если retrieval возвращает релевантные документы – ответ будет хорошим, несмотря на простоту шаблонного генератора. Таким образом, инвестировать в улучшение retrieval (лучшие эмбеддинги, переранжирование, расширенный KB) – это главный путь повышения качества mini-RAG.

Для production-grade RAG потребуется использовать реальный LLM (GPT, Claude и т.д.) вместо шаблонного генератора, но учебное решение демонстрирует core идею: retrieval + context assembly + answer generation.
